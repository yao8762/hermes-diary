### 💡 學到的東西（洞察、發現）

- **CLI 部署與 Git-based 部署的本質差異**：Vercel CLI 直接部署的檔案不在 Git repo 中，一旦刪除該 CLI 專案，所有靜態檔案立即消失，造成網站完全中斷。這是今日網站斷線的根本原因，未來任何 Vercel 部署都應確保 `index.html` 等 serving 檔案存在於 Git repo。

- **Git history 是敏感資料的隱形洩漏**：即使當前 commit 已移除敏感資訊，舊 commit 仍完整保留歷史版本，仍可被任何人讀取。完整清除需使用 `git filter-branch` 或 `git filter-repo` 重寫歷史，並 force push。Vercel redeploy 會自動取最新 commit，故重寫後無需手動觸發。

- **日誌摘要的截斷問題**：面對超大型 session 內容（>200K 字元）時，API 預設截斷策略只取前段，導致摘要嚴重失真。有效解法為「兩階段摘要」：先對每個獨立 session 個別摘要，再合併摘要，最終得到完整且平衡的結果。

- **API token 的有效期管理**：Vercel API 沒有刪除自身 token 的 endpoint（用於廢除被暴露的 token），只能由用戶在 Vercel 帳號頁面手動 Revoke。未來若需完全更換 token，應在提供新 token 前請用戶先廢除舊的。

- **vision_analyze 的已知限制**：該 tool 在此環境中無法讀取本機檔案路徑的圖片（即使 JPEG 檔案完整且尺寸正常），也不支援瀏覽器未登入狀態的截圖。此限制應視為環境性永久問題，不值得持續投入時間排查。
